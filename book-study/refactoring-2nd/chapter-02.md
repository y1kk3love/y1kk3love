# 2장 — 리팩터링 원칙

## 리팩터링이란?

소프트웨어의 겉 동작은 유지한 채, 코드를 이해하고 여러 가지 리팩터링 기법을 통해 수정하기 쉽도록 내부 구조를 변경하는 것.  
작은 단위로 진행하는 것을 권장하므로 언제든지 중지해도 소프트웨어가 작동할 수 있어야 한다.

---

## 기능 추가 vs 리팩터링 분리

- 기능 추가 시: 기존 코드를 건드리지 않고 기능 추가에만 집중
- 리팩터링 시: 코드 재구성에만 집중

---

## 리팩터링의 이유

- 소프트웨어의 설계가 좋아진다.
- 소프트웨어를 이해하기 쉬워진다.
- 버그를 찾는 것이 쉬워진다.
- 작업 속도를 높일 수 있다.
- **한마디로 개발의 지속력을 높여준다.**

---

## 리팩터링은 언제 진행해야 하는가

| 시기 | 설명 |
|------|------|
| 반복되는 문제 | 같은 문제가 3번 이상 발생하면 진행 |
| 기능 추가 직전 | 관련 기능들을 정리하여 이후 중복 작업 방지 |
| 이름 리팩터링 | 이해하기 어려운 이름을 직관적으로 수정 |
| 비효율적인 로직 | 불필요한 코드 정리 |
| 수시 리팩터링 | 작업 중 눈에 보이는 문제를 즉시 처리 |
| 계획된 리팩터링 | 비용이 높아 권장하지 않지만 안 하는 것보다는 낫다 |
| 코드 리뷰 기반 | 주기적인 리뷰를 통해 다양한 시각 확보 |

---

## 리팩터링을 진행하지 않아야 할 때

- 코드가 지저분하나 굳이 수정할 필요가 없을 때
- 새로 작성하는 것이 비용이 더 적을 때

---

## 리팩터링 시 고려할 문제

| 문제 | 내용 |
|------|------|
| 새 기능 개발 속도 저하 | 당장의 속도를 낮추지만 장기적으로는 향상. 어느 정도까지 진행할지 판단이 필요 |
| 코드 소유권 | 수정 권한이 없는 코드가 방해가 될 수 있다. 느슨한 소유권 환경이 필요 |
| 브랜치 | 브랜치 통합 비용을 줄이기 위해 개발 기간을 짧게 가져가는 편이 좋다 |
| 테스트 | 자가 테스트 코드로 리팩터링 중 문제를 빠르게 파악하고 되돌릴 수 있다 |
| 레거시 코드 | 단번에 하지 말고 단계별로 나눠 작업 |
| 데이터베이스 | 데이터 마이그레이션 스크립트로 구조적 변경을 통합 처리 |

---

## 리팩터링, 아키텍처, YAGNI

- 향후 변경에 유연하게 대처할 수 있는 유연성 메커니즘을 소프트웨어에 심어둔다.
- **현재까지 파악한 요구사항만을 해결하는 소프트웨어를 구축한다.**
- 정말 필요한 곳에만 유연성 메커니즘을 넣는다.
