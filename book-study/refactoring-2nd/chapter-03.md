# 3장 — 코드에서 나는 악취

코드 악취(Code Smell) — 리팩터링이 필요하다는 신호들.

1. **기이한 이름** — 마땅한 이름이 떠오르지 않는다면 설계에 더 근본적인 문제가 숨어 있을 가능성이 높다.

2. **중복 코드** — 같은 기능은 분리해서 각자 호출하고, 비슷한 기능은 같은 기능만 분리해서 호출하라.

3. **긴 함수** — 긴 주석이 필요할 정도라면 기능을 분리하고, 분리한 함수 이름에 의도를 담아라. 조건문이나 반복문도 독립된 함수로 만들어 관리할 수 있다.

4. **긴 매개변수 목록** — 긴 매개변수 항목들은 이해를 어렵게 만든다. 데이터 원본을 전달하거나, 함수로 포장하는 방법들로 매개변수 수를 줄이자.

5. **전역 데이터** — 전역 변수는 언제 어디서 사용할지 알 수 없다. 필요한 부분에서만 사용할 수 있도록 캡슐화하고 접근을 통제하여야 한다.

6. **가변 데이터** — 값이 변하는 데이터들은 수정을 통합하고 분리하여 언제 수정이 이루어지는지 감시하기 용이하도록 만들어야 한다.

7. **뒤엉킨 변경** — 하나의 모듈이 서로 다른 이유들로 인해 여러 가지 방식으로 변경되는 일이 많을 때 발생한다. 각자 변형해야 하는 기능이 다르다면 모듈을 분리하고, 같은 방식을 사용한다면 결합하여 관리한다.

8. **산탄총 수술** — 변경 사항이 여러 곳에 흩뿌려져 있을 때 발생한다. 변경 사항이 발생하는 대상들을 한 모듈로 묶어 관리한다.

9. **기능 편애** — 함수가 자신이 속한 모듈의 데이터보다 다른 모듈의 함수나 데이터를 호출하는 일이 잦으면 발생한다. 성향이 뚜렷하다면 자주 사용하는 곳으로 이동시키고, 일부만 그러하다면 기능을 분리하여 옮겨준다.

10. **데이터 뭉치** — 여러 곳의 데이터가 같은 영역에서 자주 호출된다면 클래스 등으로 분리하여 묶어준다.

11. **기본형 집착** — 문자열 등을 사용할 때 기본적인 형식(단위, 양식 등)을 정의하라.

12. **반복되는 switch문** — switch의 반복적인 사용을 줄여 후에 수정을 진행할 때 불필요한 작업이 발생하지 않도록 한다.

13. **반복문** — 반복문은 파이프라인으로 바꿔서 사용하라.

14. **성의 없는 요소** — 불필요하게 설계한 구조(함수가 하나뿐인 클래스, 본문과 진배없는 함수 등)를 제거하여 함수에 인라인화 한다.

15. **추측성 일반화** — 후에 사용할 것이라 생각하여 작성한 로직들(불필요한 예외처리, 테스트에서만 사용하는 함수 등)을 정리하라.

16. **임시 필드** — 사용하지 않거나 특정 상황에서만 사용하는 임시 필드들을 확인하기 쉽도록 이동시키거나 제거한다.

17. **메시지 체인** — 타고 타고 들어가서 값을 얻어와야 하는 값들은 중간 단계가 수정되면 모두 바뀌어야 하기 때문에 함수 등으로 요청하는 과정을 분리하면 수정이 용이하다.

18. **중개자** — 변수를 캡슐화한 경우 중개자를 사용하여 데이터에 접근하는데 너무 복잡하면 오히려 없애는 것이 더 도움이 된다.

19. **내부자 거래** — 모듈 사이에 데이터 전달을 줄여 결합도를 낮추고 데이터를 주고받는 과정을 투명하게 만들어 확인하기 쉽게 만들어야 한다.

20. **거대한 클래스** — 하나의 클래스가 너무 많은 작업을 진행하면 필드 수가 늘어나고 중복 코드를 유발한다. 기능들을 분리하라.

21. **서로 다른 인터페이스의 대안 클래스들** — 클래스를 바꿀 때는 인터페이스를 일치시켜라.

22. **데이터 클래스** — 데이터 저장을 목적으로 사용되는 클래스의 경우 공개 범위를 판단하여 세터 제거 등을 통해 관리한다.

23. **상속 포기** — 상속을 받아 사용하는 모듈이 불필요한 기능들까지 상속받고 있다면 서브클래스를 중간에 넣어 필요한 기능만 상속하여 해결한다.

---

**주석**  
주석을 통해 너무 많은 설명을 진행하고 있다면 리팩터링을 진행하라.
