## 1. Git-Flow

규모가 크고 릴리즈 주기가 명확한 프로젝트에 적합한 전략.  
Vincent Driessen이 제안한 방식으로, 브랜치 역할을 엄격하게 분리한다.

### 브랜치 구성

| 브랜치 | 역할 |
|--------|------|
| `main` | 실제 배포된 코드. 항상 안정적인 상태 유지. |
| `develop` | 다음 릴리즈를 위한 통합 브랜치. 개발의 기준점. |
| `feature/*` | 기능 개발용. `develop`에서 분기하고 완료 후 `develop`으로 병합. |
| `release/*` | 릴리즈 준비 브랜치. `develop`에서 분기하고 QA/버그픽스 후 `main`과 `develop`에 병합. |
| `hotfix/*` | 운영 중 발생한 긴급 버그 수정. `main`에서 분기하고 완료 후 `main`과 `develop`에 병합. |

### 흐름

```
main ──────────────────────────────────────── (배포)
        ↑ merge                        ↑ merge
develop ──────────────────────────────────────
        ↑ merge         ↑ merge
feature/A ─────       feature/B ─────
```

### 장점 / 단점

- **장점:** 브랜치 역할이 명확해 대규모 팀에서 충돌을 줄일 수 있다. 릴리즈 버전 관리에 유리하다.
- **단점:** 브랜치 수가 많아 복잡하다. 빠른 배포가 필요한 환경에는 과한 구조다.

## 2. Github-Flow

단순하고 빠른 배포 주기에 적합한 전략.  
`main` 브랜치 하나와 기능 브랜치만으로 운영된다.

### 브랜치 구성

| 브랜치 | 역할 |
|--------|------|
| `main` | 항상 배포 가능한 상태. |
| `feature/*` | 기능/버그픽스/모든 작업. `main`에서 분기하고 PR 후 `main`으로 병합. |

### 흐름

1. `main`에서 브랜치를 딴다.
2. 작업 후 PR을 올린다.
3. 코드 리뷰 후 `main`에 병합한다.
4. 즉시 배포한다.

### 장점 / 단점

- **장점:** 구조가 단순하다. CI/CD와 궁합이 좋고 빠른 배포 사이클에 적합하다.
- **단점:** 여러 버전을 동시에 유지해야 하는 환경에는 맞지 않는다.

## 3. Gitlab-Flow

Git-Flow의 복잡함과 Github-Flow의 단순함 사이의 절충안.  
환경(production, staging 등)별 브랜치를 두어 배포 흐름을 명확하게 한다.

### 브랜치 구성

| 브랜치 | 역할 |
|--------|------|
| `main` | 개발 완료된 코드. |
| `feature/*` | 기능 개발. `main`에서 분기하고 병합. |
| `staging` | 스테이징 환경 배포용. `main`에서 병합. |
| `production` | 실제 운영 환경 배포용. `staging`에서 병합. |

### 흐름

```
feature → main → staging → production
```

### 장점 / 단점

- **장점:** 환경별 배포 현황을 브랜치로 명확히 파악할 수 있다.
- **단점:** 환경 구성이 없는 소규모 프로젝트에는 불필요하게 복잡할 수 있다.

---

## 4. 전략 비교

| | Git-Flow | Github-Flow | Gitlab-Flow |
|--|----------|-------------|-------------|
| **복잡도** | 높음 | 낮음 | 중간 |
| **릴리즈 주기** | 정기 릴리즈 | 지속적 배포 | 환경별 배포 |
| **브랜치 수** | 많음 | 적음 | 중간 |
| **적합한 규모** | 중·대규모 | 소·중규모 | 중·대규모 |
| **핵심 장점** | 역할 분리 명확 | 단순·빠름 | 환경 관리 용이 |
